# Как это работает

> Технический разбор без воды — для тех, кто хочет понять, что происходит под капотом.

---

## Проблема, которую мы решаем

К 2026 году разработчики в РФ оказались под перекрёстным огнём: с одной стороны — гео-блокировки от OpenAI, Anthropic, Google («не доступен в вашем регионе»), с другой — нестабильные VPN-протоколы, которые мешают банковским приложениям и гонят весь трафик через Амстердам ради одного запроса к ChatGPT.

**aiway** решает это хирургически: `openai.com` идёт через ваш VPS, `gosuslugi.ru` — напрямую. Никаких тумблеров, никаких клиентов, работает на всех устройствах сразу.

## Два компонента

### 1. Blocky — умный DNS-сервер

Когда устройство спрашивает «где живёт `openai.com`?», обычный DNS отвечает настоящим IP-адресом серверов OpenAI в США. Мы туда не проходим — запрос режется геоблоком.

Blocky отвечает иначе:

```
Устройство:  "Где openai.com?"
Blocky:      "Вот IP" → [IP вашего VPS]

Устройство:  "Где yandex.ru?"
Blocky:      "Вот IP" → [настоящий IP Яндекса]  ← обычная резолюция
```

Для AI-доменов из нашего списка — возвращает IP вашего VPS. Для всего остального — честно резолвит через Google/Cloudflare как обычный DNS.

Результат: с точки зрения устройства, `openai.com` «живёт» на вашем VPS. Трафик пойдёт туда.

### 2. Angie — SNI Proxy

Angie принимает TLS-соединение на порту 443. Но он **не расшифровывает его**.

Вот что происходит при TLS-рукопожатии:

```
Браузер → VPS (Angie):

  TLS ClientHello {
      SNI: "api.openai.com"    ← это ОТКРЫТЫЙ ТЕКСТ, часть стандарта TLS
      ... (всё остальное зашифровано)
  }
```

Angie читает поле `SNI` — имя домена, которое браузер обязан передать открытым текстом (чтобы сервер знал, какой сертификат выдать). Это не взлом — это часть протокола TLS 1.2/1.3.

Дальше Angie:
1. Видит SNI = `api.openai.com`
2. Открывает TCP-соединение с настоящим `api.openai.com:443`
3. Начинает передавать байты туда и обратно, не вникая в содержимое

Ключевое: **Angie не является посредником в шифровании**. Он пересылает зашифрованный поток как есть. TLS-сессия устанавливается между вашим браузером и серверами OpenAI напрямую. Браузер видит оригинальный сертификат OpenAI и доволен.

## Полная схема прохождения запроса

```
1. Вы открываете chatgpt.com в браузере

2. Браузер спрашивает DNS: "где chatgpt.com?"
   └─► Blocky (ваш VPS, порт 53)
       └─► отвечает: 151.245.92.201 (IP вашего VPS)

3. Браузер открывает TCP на 151.245.92.201:443
   └─► Angie (stream блок, ssl_preread)
       └─► читает SNI = "chatgpt.com" из ClientHello
       └─► открывает TCP на настоящий chatgpt.com:443
       └─► прозрачно пересылает весь поток в обе стороны

4. TLS-рукопожатие проходит между браузером и серверами OpenAI
   (Angie видит только зашифрованные байты)

5. Браузер получает оригинальный сертификат chatgpt.com ✓
   HTTPS-замочек зелёный ✓
   Всё работает ✓
```

## Почему это не VPN

| | VPN | aiway |
|:--|:--|:--|
| Нужен клиент | на каждом устройстве | нет |
| Весь трафик через прокси | да | нет — только AI-домены |
| Банковские приложения | могут паниковать | работают нормально |
| Шифрование | доп. слой VPN поверх TLS | оригинальный TLS end-to-end |
| Батарея на телефоне | жрёт | не влияет |
| Настройка | клиент + конфиг на каждом устройстве | один раз на роутере |
| Пинг для обычных сайтов | +30-50 мс | без изменений |

## Архитектурный нюанс: конфликт stream и http на порту 443

Это нетривиальная часть. Если вы хотите одновременно:
- SNI-прокси для всего на порту 443
- HTTPS-сервер для DNS-over-HTTPS на том же порту

...наивный подход не работает. Nginx/Angie не позволяет одновременно слушать 443 в `stream {}` и в `http {}`.

**Решение aiway:**

```nginx
stream {
    # SNI map: если домен = наш DoH-домен → перебросить на внутренний порт 8443
    # всё остальное → passthrough по имени из SNI
    map $ssl_preread_server_name $upstream {
        "dns.example.com"  "127.0.0.1:8443";
        default            "$ssl_preread_server_name:443";
    }

    server {
        listen      443;
        ssl_preread on;
        proxy_pass  $upstream;
    }
}

http {
    server {
        # Только localhost — снаружи недоступен напрямую
        # Сюда попадают через stream SNI map выше
        listen 127.0.0.1:8443 ssl;
        location /dns-query { proxy_pass http://127.0.0.1:4000; }
    }
}
```

Stream-блок принимает всё на 443. Для DoH-домена — роутит на `127.0.0.1:8443` (где сидит http-сервер). Для всего остального — слепой passthrough.

## О ECH и QUIC

### ECH (Encrypted Client Hello)

ECH шифрует поле SNI. Если включён — наш прокси не видит имя домена и не может роутить. Плюс ТСПУ в РФ активно блокирует ECH-соединения.

**Отключить в Chrome:** `chrome://flags` → Encrypted Client Hello → Disabled
**Отключить в Firefox:** `about:config` → `network.dns.echconfig.enabled` = false

### QUIC (HTTP/3)

QUIC работает поверх UDP. Наш SNI-прокси работает только с TCP. Если приложение уходит на QUIC (UDP 443), оно обходит прокси и, скорее всего, получает блок.

Решение — заблокировать UDP 443 на VPS (клиент откатится на TCP):

```bash
iptables -I INPUT -p udp --dport 443 -j DROP
# Чтобы сохранить после перезагрузки:
apt-get install -y iptables-persistent
netfilter-persistent save
```

## Что aiway НЕ делает

- **Не анонимизирует** — сервер OpenAI видит IP вашего VPS
- **Не расшифровывает** — трафик остаётся end-to-end зашифрованным
- **Не скрывает факт использования AI** — домены в DNS-запросах видны (если не DoT/DoH)
- **Не обходит РКН-блокировки** — только геоблоки от самих сервисов

Это не обфускатор и не анонимайзер. Это хирургический инструмент для работы с сервисами, которые сами блокируют российские IP.
